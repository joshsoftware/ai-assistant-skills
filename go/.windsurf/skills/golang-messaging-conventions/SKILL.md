# Go Messaging & Event Queue Conventions

Conventions for producing and consuming messages in Go. **These are conventions, not a standard.**

The patterns here (idempotent consumers, DLQ, graceful shutdown, schema evolution) apply regardless of the broker. Broker-specific details are in the reference files.

## How to use this skill

1. Walk the rule categories when designing or reviewing messaging code.
2. **MUST** violations are blockers. **SHOULD** violations need a documented reason.
3. For BFSI: every message containing payment data is subject to the same PII masking, audit-trail, and data-residency rules as a database record. See `golang-bfsi-bindings` categories E, F, and M.

## Rule index

### A. Message design & schema

- **A1 [MUST]** Messages have a stable, versioned schema. Include a `schema_version` or `event_type` field in every message. Consumers must handle unknown versions gracefully (ignore or forward to DLQ, not panic).
- **A2 [MUST]** Messages are self-contained. A consumer should not need to call back to the source service to process a message. Include all data needed to act on the event.
- **A3 [MUST]** Every message has a unique `message_id` (generated by the producer with `crypto/rand`). This ID is the idempotency key for consumers.
- **A4 [MUST]** Never put PII (Aadhaar, PAN, full account number, CVV) in message payloads. Use tokenised or masked values. The message queue is not a PII-safe store. *(Binds `bfsi-india-core` rule E4.)*
- **A5 [SHOULD]** Message payloads are JSON by default. Use Protobuf or Avro for high-throughput or schema-registry use cases. Document the payload schema in the API contract.
- **A6 [SHOULD]** Include envelope metadata: `message_id`, `event_type`, `schema_version`, `source_service`, `produced_at` (UTC). The business payload is nested under a `data` field.

### B. Producer patterns

- **B1 [MUST]** Publishing a message is a side effect of a database write, not a replacement for it. Use the outbox pattern (or transactional publish) for events that must be delivered if and only if the database transaction commits.
- **B2 [MUST]** Publishing is idempotent at the producer side: publishing the same `message_id` twice should produce a single effective delivery (SQS FIFO, Kafka with idempotent producers, or consumer-side deduplication).
- **B3 [MUST]** Publishing calls use `context.Context` with an explicit timeout. A hung publish must not block a request goroutine indefinitely.
- **B4 [SHOULD]** Use batched publishing for high-throughput scenarios where individual publish-per-event latency is not required.

### C. Consumer patterns

- **C1 [MUST]** Consumers are idempotent. Processing the same message twice must produce the same result as processing it once. The broker guarantees at-least-once delivery — duplicates will arrive.
- **C2 [MUST]** Implement deduplication using the `message_id`. Store processed IDs in a fast store (Redis or DB) with a TTL spanning the expected duplicate window.
- **C3 [MUST]** Acknowledge a message only after successful processing. Never ack before processing (at-most-once delivery loses messages on crash).
- **C4 [MUST]** Consumer goroutines observe `context.Context` cancellation. A shutdown signal stops pulling new messages; in-flight messages are allowed to complete or checkpoint.
- **C5 [SHOULD]** Use a worker pool of fixed size, not an unbounded goroutine-per-message model. An unbounded model can exhaust goroutines and memory under a spike.
- **C6 [SHOULD]** Log the `message_id`, `event_type`, and consumer outcome (processed, skipped, DLQ'd) for every message. This is the audit trail for the messaging layer.

### D. Error handling, retries & DLQ

- **D1 [MUST]** Distinguish transient errors (retry) from permanent errors (DLQ). A deserialisation failure is permanent; a downstream service timeout is transient.
- **D2 [MUST]** After a configurable number of retries, route the message to a dead-letter queue (DLQ). Never drop a message silently.
- **D3 [MUST]** Monitor the DLQ. A growing DLQ is a production incident, not a background maintenance task.
- **D4 [SHOULD]** Implement exponential backoff on transient retries. For Kafka: commit the offset to a retry topic with a delay. For SQS: use visibility timeout extension. For RabbitMQ: use dead-letter exchange with a TTL.
- **D5 [SHOULD]** DLQ messages carry all original envelope metadata plus the error reason and the retry count, so operators can investigate and replay.

### E. Graceful shutdown

- **E1 [MUST]** Consumer workers stop pulling new messages when the root context is cancelled (process shutdown signal).
- **E2 [MUST]** In-flight messages have a bounded completion window — typically 30 seconds. After the window, outstanding messages are nacked (SQS/RabbitMQ) or their offsets are not committed (Kafka) so they are re-delivered to another consumer.
- **E3 [MUST]** For Kafka: do not commit offsets after the shutdown context is cancelled. Uncommitted messages are replayed by the next consumer instance.

### F. Broker abstraction

- **F1 [SHOULD]** Define a broker-agnostic interface for publishing and consuming. This allows the implementation to be swapped and simplifies unit testing.
- **F2 [SHOULD]** The consumer interface takes a handler function; the broker implementation handles connection, retry, and DLQ logic. Business logic lives in the handler.
- **F3 [MUST]** The broker interface takes `context.Context` on every publish and consume call.
