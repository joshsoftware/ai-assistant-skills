# Go Authentication & Authorisation Conventions

Conventions for authentication and authorisation in Go services. **These are conventions built on real standards.**

The underlying protocols (JWT, OAuth2, OIDC) are standardised. The *Go implementation* — how to validate a JWT safely, how to structure auth middleware, how to enforce RBAC — is engineering practice.

## How to use this skill

1. Walk the rule categories when writing or reviewing auth code.
2. **MUST** violations are blockers. For BFSI services, also check `golang-bfsi-bindings` categories A and B for the regulatory layer.
3. Never implement cryptographic primitives for signing/verification; use vetted libraries.

## Sources

- **JWT** — RFC 7519 (JWT), RFC 7515 (JWS), RFC 7517 (JWK). The JWT *format* is a standard; the Go implementation is not.
- **OAuth2** — RFC 6749 and extensions (PKCE: RFC 7636, token introspection: RFC 7662).
- **OIDC** — OpenID Connect Core 1.0 (built on OAuth2).
- **OWASP ASVS V3** (Session Management), **V4** (Access Control).

## Rule index

### A. JWT issuance & validation

- **A1 [MUST]** Use a vetted JWT library. Never write JWT parsing or signature verification by hand.
- **A2 [MUST]** Pin the expected signing algorithm before parsing. Never accept `alg: none`. Never trust the `alg` field in the token header to tell you which algorithm to use.
- **A3 [MUST]** Validate all standard claims on every token: `exp` (expiry), `nbf` (not before), `iss` (issuer), `aud` (audience). A token that passes signature verification but has an expired `exp` must be rejected.
- **A4 [MUST]** Keep access token lifetimes short: 5–15 minutes for user-facing APIs. Use refresh tokens for longer sessions.
- **A5 [MUST]** Maintain a server-side revocation mechanism. A signed JWT is valid until its `exp` even if the user logs out; a revocation list (Redis blocklist or a short-lived token family) is needed to honour logout immediately.
- **A6 [MUST]** JWT payloads are base64-encoded, not encrypted. Never put secrets, passwords, PII, or sensitive data in JWT claims unless you are using JWE (JSON Web Encryption).
- **A7 [SHOULD]** Use asymmetric signing (RS256/ES256/EdDSA) for tokens consumed by a different service or external client. Symmetric (HS256) is acceptable when the issuer and validator are the same service.

### B. Session management

- **B1 [MUST]** Session IDs are generated by `crypto/rand` — minimum 128 bits of entropy.
- **B2 [MUST]** Session cookies are set with `Secure`, `HttpOnly`, and `SameSite=Strict` (or `Lax` where cross-site is required).
- **B3 [MUST]** Rotate the session ID at authentication (login, step-up). Never reuse the pre-auth session ID post-login — this is session fixation.
- **B4 [MUST]** Logout invalidates the session server-side, not just client-side. Clearing the cookie alone leaves the session valid on the server.
- **B5 [MUST]** Enforce both an inactivity timeout and an absolute session lifetime server-side (not only via cookie `MaxAge`).

### C. OAuth2 & OIDC integration

- **C1 [MUST]** Use PKCE (RFC 7636) for all authorisation-code flows, including those in server-side web apps. PKCE is not optional even for confidential clients in modern OAuth2.
- **C2 [MUST]** Validate the `state` parameter on the OAuth2 callback to prevent CSRF.
- **C3 [MUST]** Validate OIDC ID tokens: signature, `iss`, `aud`, `exp`, `nonce`. Use the IdP's JWKS endpoint for public keys; cache the JWKS with a TTL and refresh on `kid` miss.
- **C4 [MUST]** Store OAuth2 tokens (access, refresh) as secrets — in an encrypted store or in memory only, never in plaintext browser storage. Use `HttpOnly` cookies for the web.
- **C5 [SHOULD]** Use `golang.org/x/oauth2` or an OIDC-capable library rather than implementing the OAuth2 flow by hand.

### D. Authorisation & RBAC

- **D1 [MUST]** Authorisation is always server-side. The client's claimed role, a flag in a JWT, or a query parameter saying "admin=true" is not authorisation — it is a claim that must be verified against a trusted source.
- **D2 [MUST]** Enforce least privilege. Each caller gets the minimum role needed for the operation.
- **D3 [MUST]** Check object-level entitlement on every data fetch. A valid role does not mean access to every object of that type. *(See `golang-bfsi-bindings` rule B5 — IDOR/BOLA.)*
- **D4 [SHOULD]** Model roles and permissions as data (database rows or config), not as hardcoded `if role == "admin"` branches. Data-driven RBAC can be updated without a code release.
- **D5 [SHOULD]** Log every authorisation denial with actor, resource, action, and outcome. Denials are security-relevant events.

### E. Auth middleware patterns

- **E1 [MUST]** Authentication middleware runs before every protected handler. The handler assumes the caller's identity is in the context — it does not re-parse the Authorization header.
- **E2 [MUST]** Identity is placed in the context by middleware and read from the context by handlers. Never accept identity from request parameters or body fields.
- **E3 [MUST]** A missing or invalid token returns `401 Unauthorized`. An authenticated but insufficient role returns `403 Forbidden`. Never conflate them.
- **E4 [SHOULD]** Separate authentication middleware (who are you?) from authorisation middleware (are you allowed to do this?). They have different failure modes and different reuse patterns.
- **E5 [SHOULD]** Provide a `RequireRole(role)` middleware factory that can be applied per route, rather than scattering role checks inside handlers.
